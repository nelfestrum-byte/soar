# SOAR Production Deploy (`soarctl`)

Distribution model: **build once on a machine with internet access, transfer
the bundle out-of-band, install on the air-gapped target.** No image
registry — `soarctl package` produces a single self-contained tar (all four
runtime images + compose file + config template), and `soarctl install`
never makes a network call. See
[`docs/compose/specs/2026-07-22-deploy-cli-design.md`](../../docs/compose/specs/2026-07-22-deploy-cli-design.md)
for the full design.

Single instance only — `soarctl` has no concept of multiple concurrently
managed instances. See AGENTS.md Known Limitations #9.

## 1. Build machine (has internet)

```bash
python deploy/soarctl package --version 0.9.0 --output soar-bundle-0.9.0.tar.gz
```

Builds `soar-orchestrator`/`soar-ui` images from source, pulls
`redis:7-alpine`/`postgres:16-alpine`, and saves all four plus
`deploy/prod/docker-compose.yml`, `config.yaml.template`, and `soarctl`
itself into one tarball. Transfer that file to the target machine
(USB, secure copy — outside the scope of `soarctl`).

## 2. Target machine (air-gapped)

```bash
python soarctl install soar-bundle-0.9.0.tar.gz --dir soar-prod
cd soar-prod
python soarctl doctor            # preflight: docker, ports, disk space
python soarctl init              # generates .env secrets + config.yaml — do this once
python soarctl up
python soarctl migrate --fresh   # first boot only, see note below
python soarctl users create --username admin --role admin
```

No step past `install` touches the network.

## Migrations: `--fresh` vs `--upgrade`

`init_db()`/`create_all()` runs on every orchestrator boot and only ever
creates tables that don't exist yet — it never alters an existing one.

- **New table added** (first deploy, or a migration that's purely a new
  table): `create_all()` already created it by the time you run this —
  use `soarctl migrate --fresh` (`alembic stamp head`), *not* `--upgrade`,
  or Alembic will try to `CREATE TABLE` on something that already exists
  and fail.
- **Existing table altered** (a migration adds/changes a column on a table
  that already existed): `create_all()` can't help here — use
  `soarctl migrate --upgrade` (`alembic upgrade head`).

There's no auto-detection — picking wrong can corrupt migration state, see
the deploy-cli spec's non-goals. When in doubt, check what the new
migration in `alembic/versions/` actually does.

## Day 2 operations

```bash
python soarctl status                                   # container status + /health
python soarctl logs [orchestrator|ui|redis|postgres]     # follow logs
python soarctl users create/deactivate/activate --username X [--role R]
python soarctl backup create --output backup-$(date +%F).tar.gz
python soarctl backup restore backup-2026-07-22.tar.gz --confirm
python soarctl down / up / restart
```

`backup create`/`restore` cover the Postgres database and the
`soar-data` volume (workflows/actions/connectors, including their git
history) as one archive. `restore` overwrites both — it refuses to run
without `--confirm`.

## Upgrading to a new version

1. On a build machine: `soarctl package --version X.Y.Z --output ...`,
   transfer the new bundle.
2. On the target: `soarctl install <new-bundle> --dir <same instance dir>`
   — this `docker load`s the new images and (since `.env` already exists)
   bumps only `SOAR_VERSION` in it, never `AUTH_SECRET_KEY`/
   `POSTGRES_PASSWORD` (that would lock the instance out of its own
   database) — no need to run `init` again.
3. `soarctl up` — compose recreates the containers on the new
   `SOAR_VERSION` image tag from `.env`. **Do this before `migrate`** —
   `migrate` runs `docker compose exec orchestrator ...` inside whatever
   container is currently running, so running it before `up` would exec
   into the still-running *old* container and apply the *old* image's
   `alembic/versions/`, not the new one's.
4. Check the new version's `alembic/versions/` for what changed (diff
   against what was already there before this upgrade), then
   `soarctl migrate --fresh` or `--upgrade` accordingly (see above) — only
   if the new version actually shipped a migration; skip this step
   otherwise.

## Data

- `config.yaml` / `.env` — instance-local, generated by `soarctl init`,
  never committed (see `deploy/.gitignore`)
- Named volumes (fixed names, single-instance assumption): `soar-data`,
  `soar-logs`, `soar-redis-data`, `soar-postgres-data`
