# PostgreSQL Migration — Final Report

## What Was Built

PostgreSQL support for job persistence, layered on the existing SQLite-backed
auth DB, with a single shared `database:` config section and a table-name
prefix so multiple SOAR instances can safely share one physical database.
Closes Known Limitations #3/#4 (crash recovery no-op, job history lost on
restart) when opted in — the in-memory job store remains the default, per
the spec review decision.

## Architecture

`AbstractJobStore` (mirrors the existing `AbstractJobQueue` strategy pattern):

```
AbstractJobStore (orchestrator/store/base.py)
├── InMemoryJobStore (orchestrator/store/job_store.py) — default, `JobStore` alias kept
└── SQLJobStore (orchestrator/store/sql_job_store.py) — jobs.persistence: sql
```

Table prefix (`orchestrator/db/base.py`): `configure_table_prefix()` /
`prefixed()` / `fk()`. Applied at ORM class-definition time — must run
before `orchestrator.auth.models` / `orchestrator.store.models` are
imported anywhere in the process. `main.py` now loads config and calls
`configure_table_prefix()` as the very first thing, ahead of every other
`orchestrator.*` import (previously config loaded *after* `auth.router`,
which already pulls in `auth.models`).

Alembic (`alembic/`) — versioned schema for the four tables (`users`,
`refresh_tokens`, `api_keys`, `workflow_jobs`). `env.py` reads the same
`config.yaml`/`SOAR_CONFIG` the service uses and applies the same table
prefix before importing models, so migrations can't drift from the running
app's config.

## Config

```yaml
database:
  url: sqlite+aiosqlite:///./soar.db   # or postgresql+asyncpg://...
  table_prefix: ""                      # ^[a-zA-Z0-9_]*$

jobs:
  persistence: memory   # memory | sql
```

`database:` is the single switch shared by auth and (optionally) job
persistence — no separate backend flag, no duplicated URL.

## Key Finding During Verification

The autogenerated Alembic migration hardcodes literal table/index names —
`table_prefix` never reached the DB through Alembic in the first draft, it
only affected `create_all()`. Worse, bare index names (`ix_workflow_jobs_status`)
would collide across two differently-prefixed instances sharing one Postgres
schema, since Postgres index uniqueness is per-schema, not per-table. Fixed
by hand-editing the migration to call `prefixed()`/`fk()` for every table
*and* index/constraint name at execution time, not generation time. Verified
both the unprefixed and `stage_`-prefixed cases produce correct, distinct
DDL. Documented in the migration file and in AGENTS.md: every future
migration touching these tables must follow the same convention.

A second finding surfaced only by running the real stage stack: the
orchestrator's own startup (`init_db()`/`create_all()`) already creates the
tables on first boot, so `alembic upgrade head` on a fresh DB fails with
`DuplicateTableError` — there's nothing for it to apply, the tables already
exist untracked. The correct first-deploy step is `alembic stamp head`
(mark as migrated without re-running DDL); `alembic upgrade head` is for
real future schema changes only, since `create_all()` never alters existing
tables. Added `make migrate-stamp-initial` (first deploy) vs `make migrate`
(later changes) to `deploy/stage/Makefile` and corrected AGENTS.md, which
initially documented the naive (wrong) `upgrade head`-only runbook.

A third issue was caught by the test suite itself: the initial
`test_alembic_schema.py` called `alembic.command.upgrade()` in-process,
which runs `asyncio.run()` internally and tore down the event loop this
repo's session-scoped `pytest-asyncio` fixture depends on — corrupting
every async test collected afterward (56 spurious failures). Fixed by
running `alembic upgrade head` as a subprocess in the test, matching how
it's actually invoked in production.

## Verification

- Full `tests/orchestrator/` suite: 228 passed, 1 skipped, 0 failed
- `ruff check` and `mypy --ignore-missing-imports`: clean on all
  new/changed files (pre-existing errors in untouched files — `redis_queue.py`,
  `auth/service.py`, `api/connectors.py`, `db/session.py`,
  `soar/workflows/__init__.py` — left as-is, out of scope)
- `alembic upgrade head` / `alembic downgrade base` round-trip verified on
  SQLite, both unprefixed and with `table_prefix: "stage_"`
- Real Docker Compose stack (`deploy/stage`) built and run end-to-end:
  Postgres + Redis + orchestrator + UI all healthy; ran a workflow job via
  the API; confirmed the row landed in `stage_workflow_jobs` in Postgres;
  restarted the orchestrator container; confirmed the job was still visible
  via `GET /jobs/{id}` and counted in `/status` stats — validates the
  crash-recovery fix against a real Postgres, not just SQLite unit tests
- A pre-existing `soar-redis` container (unrelated to this task, six days
  old) was inadvertently removed by `docker compose down` during cleanup
  and was recreated immediately after — its data volume (`redis-data`) was
  never removed, only the container was recreated
