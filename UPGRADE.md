# UPGRADE.md — адаптация SOAR под LLM-агентскую разработку

> Контекст: концепция проекта сместилась — LLM-агент (отдельный сервис вне SOAR)
> будет писать интеграции и воркфлоу через API SOAR прямо в живой инфраструктуре.
> Деплой не меняется (бэкенд-сервис + сторонний UI), редизайна нет. Этот документ
> фиксирует разрыв между сегодняшним API и этой ролью, и план его закрытия.
>
> Работает вместе с `AGENTS.md#known-limitations` — там технические ограничения
> движка, здесь — то, чего не хватает конкретно для агентской разработки.
> Каждый этап перед реализацией получает обычный спек/план по правилам из
> `CLAUDE.md`; этот файл — верхнеуровневая карта, не замена спекам.

## Принцип этого плана

Мы на MVP и не знаем реальный impact большинства проблем — нет ни одного
живого агента, который бы месяц писал код в SOAR. Поэтому: если решение
простое (переиспользует существующий паттерн, точечная правка) — делаем.
Если решение требует новой подсистемы или меняет архитектуру исполнения —
не делаем сейчас, а фиксируем как явный риск с условием пересмотра. Лучше
короткий список сделанного и честный список рисков, чем недоделанная
инфраструктура под гипотетическую нагрузку.

## Часть 1 — Известные проблемы

### P1. Сохранение кода не проверяется — провал молчаливый

`PUT /workflows/{name}/code`, `/actions/{name}`, `/connectors/{name}/code`
пишут файл на диск без проверки, что это вообще рабочий Python и что в нём
есть ожидаемый класс/функция. Ответ API — `{"status": "saved"}` в любом
случае. Если код не импортируется, сущность просто пропадает из реестра —
агент увидит успех и не узнает, что сломал.

**Решение:** минимальная проверка перед коммитом — код должен
компилироваться и содержать ожидаемую точку входа (класс-наследник для
workflow, одноимённая функция для action, класс-наследник для connector).
Если нет — 4xx с текстом ошибки вместо "saved". Это расширение уже
существующего обработчика PUT, не новая подсистема.

### P2. Traceback агенту недоступен

Ошибка workflow сейчас доходит до потребителя как `str(exception)` —
полный traceback существует только как текст вперемешку со stdout внутри
файла лога джобы. Агенту, чтобы понять причину падения, придётся парсить
plain-text лог вместо структурированного поля результата.

**Решение:** класть полный traceback в то же поле результата, где сейчас
лежит `str(error)` — расширение содержимого существующего контракта
(`{"error": ...}`), без новых полей и эндпоинтов.

### P3. Самоописание есть только для `soar/tools/`

`GET /tools` — единственное место, где сигнатуры и докстринги видны без
чтения исходников (через AST, без импорта). Actions, connectors и workflow
такого же read-only описания не имеют: чтобы узнать методы коннектора,
агенту нужно прочитать файл целиком — самая дорогая по контексту операция
при 25+ коннекторах.

**Решение:** тот же паттерн интроспекции, что уже работает для `/tools`,
распространить на actions и connectors; добавить docstring класса в meta
workflow. Переиспользование готового механизма, не новая инфраструктура.

### P4. Агент не может запросить "как работать с SOAR"

Из требуемого набора (системный промпт, настраиваемый пользователем
промпт, one-shot примеры, документация) сейчас нет ничего, кроме
`/tools` и одного шаблона кода на тип workflow.

**Решение (минимальный MVP-объём):** один встроенный системный промпт —
статический markdown-файл, версионируется вместе с кодом, отдаётся
read-only ручкой. Плюс один пользовательский промпт, редактируемый по
тому же CRUD+git паттерну, что уже есть для actions/workflows (не новая
механика, а ещё один файл в уже существующей схеме). Библиотеку one-shot
примеров как отдельную сущность в этот объём не включаем — это отдельная
CRUD-подсистема, а не точечная правка; смотри риск-реестр.

### P5. Код workflow импортируется в процесс оркестратора

Регистрация workflow (`GET /workflows`, планировщик) требует импорта файла
в процесс API — top-level код агента исполняется не только в изолированном
subprocess раннера, но и внутри самого оркестратора при каждом
сохранении/reload. Subprocess-изоляция (`soar/runner.py`) защищает только
запуск `run()`, не сам факт регистрации.

**Решение отсутствует в этом плане.** Правильный фикс — метаданные через
AST-разбор без импорта (как уже сделано для `/tools`) — требует
переписать discovery для workflow (schedule/type/token/path атрибуты),
это не точечная правка, а изменение работающего механизма с риском
регрессий. **Принято как риск** — см. реестр рисков.

### P6. Конфиг коннектора отдаёт секреты в открытом виде

`GET /connectors/{name}/config` возвращает пароли/токены как есть. Если
агенту нужно читать конфиг, чтобы, например, понять текущие параметры
интеграции, секрет уйдёт в контекст LLM.

**Решение отсутствует в этом плане.** Маскирование секретов при чтении с
корректным merge на записи (не путать "не менялось" и "очистить") —
не тривиальная логика для MVP. **Принято как риск**, компенсируется
минимально: агент не получает роль выше необходимой (см. P7) — доступа к
управлению пользователями/ключами/аудитом у него не будет в любом случае.

### P7. Право писать код = admin целиком

Сейчас `PUT` кода workflow/action/connector требует роли `admin` — той же,
что нужна для управления пользователями, API-ключами и просмотра audit-log.
Чтобы дать агенту возможность писать код, ему пришлось бы выдать полный
административный доступ.

**Решение:** роль в системе — обычная строка без фиксированного enum,
добавление новой роли не требует миграции схемы. Ввести отдельную роль
для агента с правом на код и jobs, но без доступа к auth-администрированию
и audit-log. Точечное расширение существующей RBAC-модели.

### P8. История и откат правок недоступны через API

`GitManager` уже умеет `history()`, `get_content()`, `diff()`, `restore()`
— вся механика для "агент сломал файл, надо посмотреть прошлые версии и
откатиться" готова, но не выставлена наружу ни одной HTTP-ручкой.

**Решение:** тонкая read+admin-write обёртка над уже существующими
методами `GitManager` для workflows/actions/connectors. Инфраструктура
полностью готова, это самый дешёвый пункт плана.

### P9. Webhook-токен нестабилен при частых правках

Токен webhook-workflow по умолчанию генерируется как атрибут класса
(`secrets.token_urlsafe()` в шаблоне), а не хранится отдельно. При
пересборке реестра после правки код исполняется заново, и токен может
незапланированно замениться. Раньше это был редкий сценарий (человек
правит код нечасто), но при агентской итеративной разработке — правки
будут частыми, и внешняя система, знающая старый токен, перестанет
приниматься.

**Решение:** при перезагрузке meta предпочитать токен, уже сохранённый в
`orchestrator_state.yaml` — том же runtime-файле, что уже хранит
enabled/disabled по каждому workflow (`_save_state`/`_remove_from_state`
в `orchestrator/api/workflows.py`, чтение — в `load_workflow_metas`), а не
значение из класса. Файл не коммитится в git и не требует миграции — но
формат значения на webhook-workflow придётся сменить со строки
(`"enabled"`/`"disabled"`) на объект (`{enabled, token}`), это правка
формата существующего файла, а не просто новое поле; отражать в плане
этапа 1 отдельным пунктом, не как побочный эффект.

### P10. Нет блокировки одновременного редактирования

Два актора (агент и человек, или два агента), правящие один файл через
`PUT`, перезаписывают друг друга без предупреждения — last-write-wins.

**Решение отсутствует в этом плане.** Optimistic locking (условные
заголовки, обработка конфликтов) — усложнение ради сценария, вероятность
и частота которого при одном агенте-исполнителе пока неизвестна.
**Принято как риск** — после P8 хотя бы есть путь восстановления
постфактум через git-историю.

### P11. Логи workflow — неструктурированный текст

Лог джобы — это stdout вперемешку со всем, что печатают actions и
connectors, без разбивки на структурированные записи и без единого
формата для программного разбора агентом.

**Решение отсутствует в этом плане.** Полноценная структурированная
трассировка — изменение контракта логирования во всех сущностях сразу,
заметный объём работы. Базовый log-per-job (P2 закрывает главный пробел —
доступ к traceback) уже даёт минимально достаточную прозрачность для
MVP. **Принято как риск.**

## Часть 2 — Этапы

Каждый этап закрывает проблемы из части 1 и оставляет реестр рисков
нетронутым до отдельного решения его пересмотреть. Порядок — по
зависимости: этап 1 должен быть сделан раньше остальных, иначе агент
не сможет надёжно проверить, что он вообще исправно пишет код.

### Этап 1 — Замкнуть цикл разработки агента

**Цель:** агент, написавший код через API, должен без чтения серверных
логов узнать, что код принят/отклонён, почему упал запуск, и иметь путь
назад, если сломал рабочую версию.

**Закрывает:** P1 (проверка при сохранении), P2 (traceback в результате),
P8 (история/откат через API), P9 (стабильный webhook-токен).

**Сознательно не делаем на этом этапе:** ничего из самоописания (это
этап 2) и ничего из прав доступа (этап 3) — агент на этом этапе может
работать под существующей ролью `admin`, чтобы не блокировать цикл
разработки ожиданием RBAC-изменений.

**Готово, когда:** сохранение заведомо неисполняемого кода возвращает
понятную ошибку, а не тихий 200; упавший job отдаёт полный traceback;
для любого редактируемого файла можно посмотреть историю версий и
откатиться через API, не заходя на сервер.

### Этап 2 — Система объясняет себя

**Цель:** агент может узнать, что доступно в SOAR (примитивы, действия,
конвенции), не читая исходный код и не полагаясь на внешнюю "бумажную"
документацию.

**Закрывает:** P3 (интроспекция actions/connectors), P4 (системный +
пользовательский промпт, в минимальном объёме — без библиотеки примеров).

**Сознательно не делаем на этом этапе:** библиотеку one-shot примеров как
отдельную редактируемую сущность — откладываем до тех пор, пока не станет
понятно, что статического промпта и интроспекции реально не хватает.

**Готово, когда:** по любому action/connector/workflow можно получить
сигнатуры и докстринг read-only запросом; агент может запросить системный
промпт и (если задан) пользовательский промпт одним вызовом каждый.

### Этап 3 — Права доступа под агента

**Цель:** агент работает не под `admin`, а под ролью, дающей ровно то,
что нужно для разработки и запуска (код, jobs), без доступа к
администрированию пользователей/ключей/аудита.

**Закрывает:** P7 (отдельная роль).

**Сознательно не делаем на этом этапе:** маскирование секретов (P6) и
ограничение по конкретным ресурсам внутри роли (например, "агенту нельзя
трогать connector X") — при одной плоской роли это не нужно, пока не
появится сценарий с несколькими агентами разного доверия.

**Готово, когда:** для агента можно выпустить API-ключ/пользователя с
правом писать код и запускать jobs, но без прав на `/auth/keys` и
`/audit-log`.

## Часть 3 — Реестр принятых рисков

| # | Риск | Почему не чиним сейчас | Когда пересмотреть |
|---|------|------------------------|---------------------|
| P5 | Код workflow импортируется в процесс оркестратора при сохранении/reload — top-level код агента исполняется вне subprocess-изоляции | Фикс требует переписать discovery без импорта (AST), риск регрессий выше пользы на MVP | Если появится инцидент, или объём агентских правок вырастет настолько, что риск перестанет быть теоретическим |
| P6 | Секреты коннектора отдаются агенту в открытом виде через `GET /connectors/{name}/config` | Маскирование с корректным merge на записи — не точечная правка | Если агенту реально понадобится читать (не только писать) конфиги с секретами в рабочем процессе |
| P10 | Нет блокировки параллельного редактирования одного файла | Неизвестна частота конфликтов при одном агенте-исполнителе; git-история (этап 1) даёт recovery | Если появится второй одновременно пишущий актор (второй агент или агент + активно правящий человек) |
| P11 | Лог workflow — неструктурированный текст, не программно разбираемый | Traceback (этап 1) закрывает главный пробел; полная трассировка — отдельная тема с большим объёмом | Если агенту потребуется автоматически разбирать поведение workflow по логам, не только факт успеха/ошибки |
| P4-примеры | Библиотека one-shot примеров отложена за пределы этапа 2 | Отдельная CRUD-подсистема, не точечная правка; неизвестно, нужна ли она при наличии промпта и интроспекции | Если статического промпта и `/tools`-подобной интроспекции агенту окажется недостаточно на практике |

## Не в скоупе этого плана

- Sandboxing subprocess сверх того, что уже есть (allowlist env, отдельный
  процесс на джобу) — не требование текущей концепции, требование "агент
  пишет код" не равно "агент пишет вредоносный код", доверие к агенту как
  к актору уже заложено в идее.
- Multi-agent координация (несколько агентов одновременно работают над
  одной инфраструктурой) — не описано в исходной концепции, отдельная
  задача при появлении реального сценария.
- Любой staging/preview-контур перед применением изменений — концепция
  явно описывает работу "на месте, в живой инфраструктуре"; enable/disable
  и dry-run (уже есть в проекте) — тот минимальный контроль, который не
  меняет модель деплоя.
